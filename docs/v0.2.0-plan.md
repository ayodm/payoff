# payoff v0.2.0 — Implementation Plan

> Produced by a planning pass after `claude-time` v0.1.2 shipped, under
> the project's prior name (`claude-time`). The plan executed as written;
> the project was renamed to `payoff` at v0.2.0 cut to put functional
> meaning before tool branding. Historical references to `claude-time`
> in this doc are left as-is.

## Goal

Move from *outcome-only* tracking to *driver capture + correlation*. Answer:
"does adding a skill / rewriting CLAUDE.md / changing prompt style move my
retention number, and where?"

The v0.1.x machinery already captures outcomes (retention, cost, quadrant,
per-file waste). What's missing is the *independent variable* — what was in
the prompt environment when the session ran — and an analysis layer that
joins driver to outcome across many sessions.

## 1. Capture-surface inventory

What's reachable, and at what cost. Each row below is a *driver* claude-time
should record for correlation.

| Driver | Where it lives | Cheapest capture point | Schema cost |
|---|---|---|---|
| Active skills | `~/.claude/skills/*/SKILL.md` + per-project `.claude/skills/`. Also plugin-bundled skills under `~/.claude/plugins/marketplaces/*/skills/`. Plugin enable state in `settings.json::enabledPlugins`. | Filesystem walk inside `SessionStart` hook | `Vec<String>` of skill names + SHA256 of each SKILL.md content |
| CLAUDE.md hash | `<cwd>/CLAUDE.md` (project), `<cwd>/.claude/CLAUDE.md`, `~/.claude/CLAUDE.md` (user). Walk up from cwd. | Filesystem read inside `SessionStart` | `BTreeMap<String, String>` of `relpath → sha256` |
| Active hooks | `~/.claude/settings.json::hooks` (already manipulated in `install.rs`) | Read inside `SessionStart` | `Vec<String>` of hook event names + per-event command count |
| Model used | Already in payload, already on `SessionRecord::model`, **never reported** | Already captured | None — just surface in reports |
| Cache hit ratio | `cache_read_tokens / (cache_read_tokens + cache_creation_tokens + input_tokens)`. Already captured. | Already captured | None — computed property |
| Initial prompt characteristics | First `role: user` message in the transcript JSONL. Already parsed in `transcript.rs`. | Extend `TranscriptStats` to record first-user-message text, derive flags during parse | `PromptShape` sub-struct on `SessionRecord` |
| Plan-mode | Plan-mode is a Claude Code session attribute. **Not currently surfaced in the SessionStart payload.** | Capture whatever extra keys appear in SessionStart payload into a `BTreeMap<String, Value>` "extras" bag | One bag field |
| `UserPromptSubmit` per-turn prompts | Requires a third hook. Higher cost; defer to Phase 3. | New hook command `claude-time hook user-prompt-submit` | New event type, new write path |

### Not obtainable without Claude Code changes

- **Why** a skill was invoked (just that it exists in the directory).
- **Which** skill Claude actually consulted vs. which were merely available.
- **Subagent** identity per turn.

These are interesting but out of scope; call out in `## What this report does NOT measure`.

## 2. Phased plan

Five phases, ordered by cost/value. Phases 0–2 are the v0.2.0 floor; 3–4 are stretch.

### Phase 0 — Cache hit ratio + model surfacing (free wins)

**Why first:** zero new capture, zero new fields, no hook changes.

- **Files touched:** `src/model.rs` (one method), `src/html_report.rs`, `src/report.rs`, `src/serve.rs::session_fragment`
- **New code:** `impl SessionRecord` — `pub fn cache_hit_ratio(&self) -> Option<f64>` computed from existing fields. Returns `None` when denominator is zero.
- **Report changes:** add "Cache hit ratio" + "Sessions by model" rows to `render_totals` (HTML + markdown); add model + cache-hit-ratio to `serve::session_fragment` detail grid.
- **Test strategy:** unit test on `cache_hit_ratio` for (a) zero denominator → None, (b) all-cache-read → 1.0, (c) mixed; HTML smoke test for new strings.
- **User-visible:** "report now shows cache hit ratio and which model spent your money."

### Phase 1 — Driver capture: skills, CLAUDE.md, hooks (SessionStart side)

- **New module:** `src/env_capture.rs`
  - `pub fn capture(cwd: &Path) -> CapturedEnv`
  - Reads: `~/.claude/settings.json::enabledPlugins`, `hooks` event names (excluding our own commands), `~/.claude/skills/*/SKILL.md`, project `.claude/skills/`, plugin-bundled skills, `<cwd>/CLAUDE.md` (walk up to repo root, depth 8), `~/.claude/CLAUDE.md`.
  - Hashing: `git2::Oid::hash_object` (already a dep) if it accepts arbitrary bytes, else hand-rolled FNV-1a 64-bit (~20 lines). No new crate.
- **`SessionRecord` extensions** (all `#[serde(default, skip_serializing_if)]`):
  ```rust
  pub active_skills: Vec<SkillRef>,
  pub claude_md_files: BTreeMap<String, String>,
  pub active_hook_events: BTreeMap<String, u32>,
  pub enabled_plugins: Vec<String>,
  pub env_extras: BTreeMap<String, Value>,
  ```
  Plus `pub struct SkillRef { pub name: String, pub content_hash: String, pub source: SkillSource }` where `SkillSource = User | Project | Plugin(String)`.
- **Hook wiring:** `src/hooks.rs::populate_session_start` calls `env_capture::capture(cwd)`. Wrapped in own error catch so a failed capture doesn't taint existing capture.
- **Test strategy:**
  - `env_capture` unit tests against tempdir with synthetic `settings.json` / `CLAUDE.md` / fake skill dirs.
  - **Self-exclusion test:** claude-time's own hooks NOT counted in `active_hook_events`.
  - **Cross-version parse test:** v0.1.2-shaped session JSON deserializes with empty defaults.
- **User-visible:** "your session records now know which CLAUDE.md, skills, and hooks were in scope. No reports surface them yet — Phase 2 does that."

### Phase 2 — Correlation views in the report

- **New module:** `src/correlate.rs`
  - `pub struct DriverGroup { key: DriverKey, sessions: Vec<&SessionRecord>, avg_retention: f64, avg_cost: f64, n: usize }`
  - `pub enum DriverKey { Skill(String), ClaudeMdHash(String), HookEvent(String), Model(String), CacheBucket, PromptFlag, ToolSequence(Vec<String>), EditWithoutRead }`
  - `pub fn group_by_driver(sessions: &[SessionRecord]) -> Vec<DriverGroup>` — partitions and computes deltas vs. all-sessions baseline.
  - **Sub-decision:** add `tool_sequence: Vec<String>` to `SessionRecord` (capped at first 200 calls) populated in `transcript.rs::ingest_entry`. Needed for "Edit without prior Read" + N-gram pattern mining.
- **New HTML section "Drivers"** — between "Where time was wasted" and "Quadrant". One subsection per driver type, table columns: `Group | N | Avg retention | Δ vs. baseline | Avg cost | Δ vs. baseline`. Only render groups with `n >= 3` so single-session noise doesn't dominate.
- **New HTMX endpoint:** `GET /driver/{type}/{key}` → fragment listing sessions in that group (reuses existing `format_session_row`).
- **Test strategy:** unit tests on `group_by_driver` with controlled session set; HTML snapshot tests (`insta`, already a dev-dep, currently unused — good time to lock in); integration test seeding two CLAUDE.md hashes and asserting the Drivers section contains both group rows.
- **User-visible:** "report groups sessions by skill, CLAUDE.md hash, and hook config — see which environment changes moved retention. Click to drill in."

### Phase 3 — `UserPromptSubmit` hook (stretch)

- **New CLI variant:** `Command::Hook { event: HookEvent::UserPromptSubmit }`. Extend enum + dispatch.
- **New hook handler:** `populate_user_prompt_submit(record, &payload)` reads `payload.prompt`, computes `PromptShape { length, has_code_block, has_question_mark, looks_like_command, looks_like_question }`, pushes onto `SessionRecord::prompts: Vec<PromptShape>` (cap at 50).
- **Install/uninstall:** extend `HOOK_EVENTS` in `install.rs`. Migration story already exists. **Bump legacy-migration test counts.**
- **Plugin manifest:** add the third event to `hooks/hooks.json` and `.claude-plugin/plugin.json`.
- **Report:** prompt-shape histograms in Drivers section.
- **User-visible:** "claude-time learns from your first words — see whether questions outperform commands."

### Phase 4 — Cache buckets + tool-sequence pattern mining (stretch)

- **Cache-bucket driver:** `CacheBucket::{Cold(<30%), Warm(30..70), Hot(>=70)}` partitions sessions by cache hit ratio.
- **Tool-sequence mining:** 3-grams in `tool_sequence`. Top-K (K=10) most-frequent → retention delta. Useful patterns surface organically.
- **User-visible:** "warm-cache sessions retain 2x; `Read → Edit` outperforms cold `Edit` by N%."

## 3. Schema migration discipline

- Every new field uses `#[serde(default, skip_serializing_if = "...")]` — matches existing pattern, v0.1.2 records parse without intervention.
- **No `schema_version` field.** Existing fail-soft load swallows parse errors and falls back; missing-field defaults give zero-cost forward compat.
- archive.jsonl gains new fields per line, but JSONL is line-independent — old lines stay parseable.
- `env_capture` runs at SessionStart only. Sessions started under v0.1.2 and ending under v0.2.0 will have empty driver fields — they don't participate in correlation (filtered by `n >= 3`).

## 4. Storage rule compliance

- All new data fits inside existing `SessionRecord`. No new file types in `sessions/`. Archive rollup handles uniformly.
- Worst-case record growth: ~5 skill names @ 40 bytes + 32-byte hashes, ~3 CLAUDE.md hashes, ~5 hook events, ~200-entry tool sequence ≈ **1.5 KB extra per session**. ~7x growth on README's 250-byte baseline = ~35 MB/year for a heavy user.

## 5. Fail-soft discipline

- `env_capture::capture` returns `CapturedEnv` with `Default` — any read failure produces empty data, never an error. `eprintln!` on internal errors (matches `hooks.rs`).
- All new hook handlers route through existing `run` wrapper — same fail-soft guarantee.

## 6. No-new-dep audit

| Need | Existing crate | Notes |
|---|---|---|
| Content hashing | `git2::Oid::hash_object` or hand-rolled FNV-1a | FNV-1a in ~20 lines is sufficient — these are grouping keys, not crypto |
| Directory walk | `std::fs::read_dir` + bounded recursion | No `walkdir` |
| Settings parse | `serde_json::Value` (already in install.rs) | Yes |
| Snapshot testing | `insta` (already dev-dep, unused) | First real usage |

**Verdict:** no `cargo add` needed.

## 7. Open questions / risks

1. **`git2::Oid::hash_object` API surface** — needs confirmation it accepts `&[u8]`. Fallback: FNV-1a. Don't block.
2. **Plan-mode visibility in SessionStart payload** — current `populate_from_payload` only reads `cwd`, `transcript_path`, `model`. Recommend dumping full payload once on first run (env-gated, not committed) to inventory what arrives. `env_extras` bag captures unknown keys forward-compatibly.
3. **Skill discovery completeness** — plugin-bundled skills under `~/.claude/plugins/marketplaces/<mp>/<plugin>/skills/`. Cross-reference with `settings.json::enabledPlugins` (format `"<plugin>@<marketplace>"`).
4. **`UserPromptSubmit` payload key for prompt text** — unverified. Best-effort `get("prompt").or(get("user_prompt"))`. If absent, defer Phase 3.
5. **HTMX URL space** — `/driver/{type}/{key}` may collide with skill names containing slashes/unicode. Percent-encode driver keys on render.
6. **Excluding our own observer** — `active_hook_events` would include claude-time's own hooks. Filter `claude-time hook *` commands and (configurable) the project named `claude-time` itself. Otherwise the correlation layer "discovers" the meaningless "every session has SessionStart" pattern.

## 8. Explicit scope cuts (rejected for v0.2.0)

- **zstd compression of archive.jsonl** — adds dep, fights "no new deps" rule. Defer to v0.3.
- **Per-turn cost attribution** — too speculative.
- **MCP server wrapper** — HTMX server already answers "where am I wasting time."
- **Causal claims (skill X *caused* +20% retention)** — needs controlled comparison. Naive correlation in Phase 2 surfaces patterns; footer caveat ("correlation, not causation — pin a CLAUDE.md hash and toggle one skill to compare") is the right v0.2 honesty.
- **Real-time skill usage detection** — we capture availability, not usage. v0.3 candidate.
- **Sentiment / satisfaction prompts** — v0.1 footer explicitly says no thumbs. Stays out.
- **HTTP egress for shared baselines** — no telemetry. Stays out.
- **A `schema_version` field** — `#[serde(default)]` doing the job.
